# ADR-007

Title:
FluentValidation for Input Validation

Status:
Accepted

Date:
2026-07-05

Owner:
Mohamed Eleraki (Project Owner)

---

## CONTEXT

Input validation for commands and queries needs to be consistent, expressive, and decoupled from the handlers themselves.

## DECISION

We will use **FluentValidation** for validating commands and queries before they reach handlers.

## RATIONALE

1. **Separation of Concerns:** Validation rules live outside handlers
2. **Fluent API:** Expressive, readable rule definitions
3. **Testability:** Validators can be unit tested independently
4. **Reusability:** Rules can be shared across multiple handlers
5. **Integration:** Works seamlessly with MediatR pipeline

## CONSEQUENCES

### Positive
- Validation logic is isolated and maintainable
- Consistent error messages across the platform
- Easier to evolve validation rules over time
- Reduced boilerplate in handlers

### Negative
- Additional dependency and configuration
- Requires developers to learn FluentValidation syntax
- Overhead for very simple validation scenarios
