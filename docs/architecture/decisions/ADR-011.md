# ADR-011

Title:
Repository Pattern

Status:
Accepted

Date:
2026-07-05

Owner:
Mohamed Eleraki (Project Owner)

---

## CONTEXT

Data access logic risks scattering across the codebase, making it difficult to maintain and test.

## DECISION

We will use the **Repository Pattern** to abstract data access and provide a collection-like interface for aggregates.

## RATIONALE

1. **Abstraction:** Hides EF Core details from the domain layer
2. **Testability:** Easy to mock repositories in unit tests
3. **Consistency:** Standardized CRUD interface across all aggregates
4. **Flexibility:** Data access implementation can change without affecting domain logic
5. **Intent:** Code expresses what it needs, not how to get it

## CONSEQUENCES

### Positive
- Domain layer remains independent of EF Core
- Simplified unit testing with in-memory or mock repositories
- Consistent data access patterns across the codebase
- Easier to enforce business rules at the repository boundary

### Negative
- Additional abstraction may seem unnecessary for simple queries
- Requires discipline to avoid leaking query logic into repositories
- Generic repositories can become leaky abstractions
- More interfaces and classes to maintain
