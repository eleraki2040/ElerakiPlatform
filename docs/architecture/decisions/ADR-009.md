# ADR-009

Title:
Strongly Typed IDs

Status:
Accepted

Date:
2026-07-05

Owner:
Mohamed Eleraki (Project Owner)

---

## CONTEXT

Using primitive types like `Guid` or `int` for entity identifiers leads to errors where the wrong ID type is passed between operations.

## DECISION

We will use **strongly-typed IDs** for all domain entity identifiers instead of primitive types.

## RATIONALE

1. **Type Safety:** Prevents mixing up different entity IDs at compile time
2. **Readability:** Code expresses intent clearly (e.g., `ProductId` vs `OrderId`)
3. **Domain Modeling:** IDs become first-class domain concepts
4. **Refactoring:** Safer changes when entity boundaries evolve
5. **Validation:** Easier to validate input at the type level

## CONSEQUENCES

### Positive
- Compile-time protection against ID mix-ups
- More expressive and self-documenting code
- Reduced runtime errors from incorrect ID usage
- Better alignment with Domain-Driven Design principles

### Negative
- Additional boilerplate for ID wrapper types
- Slight learning curve for new developers
- May require custom value converters for EF Core
- Serialization and JSON handling needs consideration
