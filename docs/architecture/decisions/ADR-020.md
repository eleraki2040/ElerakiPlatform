# ADR-020

Title:
Serilog for Logging

Status:
Accepted

Date:
2026-07-05

Owner:
Mohamed Eleraki (Project Owner)

---

## CONTEXT

The platform needs a structured logging solution that provides rich context for debugging, monitoring, and auditing.

## DECISION

We will use **Serilog** as the primary logging library across all platform components.

## RATIONALE

1. **Structured Logging:** Logs are emitted as structured data rather than plain text
2. **Flexibility:** Supports multiple sinks including console, file, and Seq
3. **Performance:** Efficient message template parsing and minimal overhead
4. **Enrichment:** Easy to add contextual information like user ID and request ID
5. **Ecosystem:** Rich set of sinks and integrations for .NET applications

## CONSEQUENCES

### Positive
- Logs are machine-parseable and searchable
- Easier to correlate events across distributed operations
- Flexible configuration without code changes
- Strong community support and documentation

### Negative
- Requires learning Serilog-specific configuration and sinks
- Structured logging requires discipline to log consistently
- Additional configuration complexity for sink setup
- Message templates must be maintained carefully
